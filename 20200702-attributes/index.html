Attributes

What are attributes?

  • provide additional information to the compiler
  • may influence code generation
  • may influence compiler warnings
  • do not change the meaning of the program

Compiler-dependent syntax

  • MSVC
__declspec(attribute-name)
  • GCC / Clang
__attribute__((attribute-name))

Unified syntax

Introduced in C++11 

[[attribute-name]]

Unified syntax

Also, multiple attributes in a single statement are possible: 

[[attribute1, attribute2]]

Unified syntax

Compiler-specific attributes are namespaced, e.g. for GCC: 

[[gnu::always_inline]]

Attributes added to the standard

C++11

[[noreturn]]

[[noreturn]] void terminate() {
  // ...
}

Indicates that function traps control flow somehow and will not return from a call.

[[carries_dependency]]

void foo(int& [[carries_dependency]] value);

Indicates that logic related to synchronization primitives used e.g. by atomic operation should carry over into the function call.

C++14

[[deprecated]]

Also [[deprecated(reason)]]. Used to indicate that a class, function, enum, variable, typedef or function parameter should not be used and may be removed in future. Generates a compiler warning.

C++17

[[nodiscard]]

Used to indicate that ignoring a value returned by a function should not be ignored. Generates a compiler warning with -Wunused-result (enabled by default in GCC 10). 

[[nodiscard]] int foo();

This attribute can also be applied to classes and class constuctors, although the behaviour is not consistent in this case.

[[fallthrough]]

With compiler warning enabled, all missing break statements in switch statements are reported by the compiler. By writting [[fallthrough]] at the position a break statement would otherwise be, we inform the compiler that this is what we want.

This is NOT a fallthrough: 

void do_stuff(int param) {
  switch(param) {
  case 0: // grouping labels is NOT a fall-
  default: // -through
    foo();
  }
}

[[fallthrough]]

void do_stuff(int param) {
  switch (param) {
  case 0:
    extra_foo(); // THIS IS A FALLTHROUG
  default:
    foo();
  }
}

[[fallthrough]]

void do_stuff(int param) {
  switch (param) {
  case 0;
    extra_foo();
    [[fallthrough]]; // This makes your compiler happy!
  default:
    foo();
}
}

[[maybe_unused]]

A proper way to silence an “unused” warning from the compiler - usually for variables or unused functions. 

[[maybe_unused]] std::string debug_msg = /* some debug info */;
DEBUG_PRINT(debug_msg);

C++20

[[nodiscard(reason)]]

Same as for C++17, but compiler will probably print the reason as part of the warning.

[[likely]] / [[unlikely]]

Marks a block of code as likely or unlikely to be executed. Provides a hint to the optimizer. 

if (very_likey_to_be_TRUE_in_most_cases) [[likely]] {
  // handle likely case
} else {
  // the unlikely case
}

if (very_likely_to_be_FALSE_in_most_cases) {
  // the rare true case
} else [[likely]] {
  // the more common false case
}

if (very_unlikely_error_condition) [[unlikely]] {
  // handle error
}

[[no_unique_address]]

Allows for optimizing object size in specialized cases. Stateless allocators are mentioned as a major use case. 

class Empty {};
struct S1 {
    int x;
    [[no_unique_address]] Empty e;
};
struct S2 { int x; };
static_assert(sizeof(S1) == sizeof(S2));

Checking attribute support

GCC and clang provide a built-in macro for detecting attribute support. 

#if __has_cpp_attribute(nodiscard)
#define NODISCARD [[nodiscard]]
#else
#define NODISCARD
#endif
