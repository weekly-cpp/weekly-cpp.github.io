The Good Old if

New additions

  • init expression

  • if constexpr

    In the C++17 standard, our good old if statement received two usability enhancementes. These are the initialization statements and the ability to mark if to be constexpr. We’ll discuss what this means exactly in a second, but first lets look at how these fit into the syntax of this well known statement.

What changed in the syntax?

Pre C++17 if syntax

if ( condition )
  statement-true
else
  statement-false

This is the if statement we all know and love (?). We have the if keyword followed by the condition in parenthesis, followed by the statement-true block, optionally followed by the else keyword in turn followed by the statement false block. When the condition is true then the statement-true gets executed. When the condition is false, the statement-false gets executed if present - or otherwise nothing is executed. Hopefully no surprises here.

Since C++17

if constexpr[opt] (init-statement;[opt] condition )
  statement-true
else
  statement-false

The [opt] marker is not part of C++, just marks optional syntax.

Now, the C++17 standard introduces two new elements to this picture. First is that between the if keyword and the condition we can now place the constexpr keyword, making the if evaluated at compile time. Second is the initialization statement, placed inside the parenthesis of the condition and separated from the condition by a semicolon. Both of this new elements are optional - since backwards compatibility is paramount in C++ and it just wouldn’t be very well received by the users if all of a sudden all the existing if statements in legacy code were broken. In any case, let’s look at the init statements in more detail now.

init-statement

if (init-statement; condition)
  statement-true;
else
  statement-false;
// is equivalent to
{
  init-statement;
  if (condition)
    statement-true;
  else
    statement-false;
}

This two are actually defined to be equivalent in the standard, there is no surprise twist here as with structured bindings.

Basic example

std::pair<Value, bool> MaybeGetSomeValue();

if (auto result = MaybeGetSomeValue(); result.second == true) {
  // do something with value
}

Structuded bindings can help

std::pair<Value, bool> MaybeGetSomeValue();

if (auto [value, success] = MaybeGetSomeValue(); success) {
  // do stuff with value
}

std::map::insert

DISCLAMER: not endorsing std::map 

std::map<K, V> m;
if (auto [element_iter, success] = m.insert({k, v}); success) {
  // element_iter points to a new element
} else {
  // element_iter points to pre-existing element with key k
}

Example use-case: Helper variable

if (int number; std::cin >> number) {
  std::cout << "You picked: " << number << '\n';
} else {
  std::cout << "That's not a number!\n";
}

Example use-case: narrowing scope of a RAII helper

std::mutex m;
// ...
// a lot of stuff in-between
// ...
if (std::lock_guard lock{m}; some_condition) {
  // do some important stuff with shared resource
}

if constexpr

  • limited usability outside of templates
  • make template programming closer to ’normal’ programming

get without if constexpr

class Vector2D {
  double x_;
  double y_;
  // ...
  public:
  template <std::size_t idx> double get();
};
template<> double get<0>() { return x_; }
template<> double get<1>() { return y_; }

Using e.g. get<2>() is a linker error (undefined reference to function).

if constexpr to the rescue

class Vector2D {
  double x_;
  double y_;
  public:
  template <std::size_t idx> double get() {
    if constexpr (idx == 0) return x_;
    if constexpr (idx == 1) return y_;
  }
};

Using e.g. get<2>() is a compilation error (no return statement).

There are a few technical nuances to the if constexpr, but I hope this brief introduction gives you at least some idea about how it can be useful.
